How to use this booklet

The companion sprint-plan PDF gave you the strategic view: timeline, themes, and a one-line description of what each Friday demo will show. This booklet goes one level deeper. For every sprint you get three things:

  • What we're building this sprint — the concrete feature list, grouped by Mobile, Backend, and Admin.
  • The Friday demo will confirm — an acceptance checklist you can tick off while watching the demo recording.
  • Deferred from this sprint — what someone might reasonably expect to be in this sprint but is actually landing later, with the destination sprint named.

The feature names use the same wording as the original feature inventory, so cross-referencing is straightforward. Anything originally marked POST (post-MVP) in the inventory is listed in the appendix at the back, not in any sprint.

Foundation: monorepo, schema, security, AI tooling, plan

Mon 1 Jun — Fri 19 Jun 2026

Sprint 0 isn't a feature-build sprint. It's the foundation that makes Sprints 1-12 possible. Everything below is already complete and reviewable in the project repository.

Repository & tooling

  • Database schema with nine core models (users, jobs, bids, reviews, messages, payments, audit log, categories, country)
  • Brand theme locked from prior prototype, modernised for Material 3 design language
  • Local-development Docker setup for Postgres and Redis
Architecture & planning
  • Eight architecture decision records covering: monorepo & stack, database conventions, multi-country readiness, category types, tasker verification model, auth tokens, edge security, OTP strategy
  • Twenty IT-audit documents (privacy policy, data flow diagrams, DSR process, incident response, encryption policy, vendor list, etc.)

  • Six custom AI-assistant tools written for the development workflow (Stripe payments, AU tax, vector matching, dispute mediator, multimodal extraction, security review)
  • Five-layer security stack in place: AI-assistant checklist, pre-commit hooks, server-side CI (lint, typecheck, tests, secrets scan, static analysis), branch protection ruleset, planned runtime defences

Sprint 1

Backend authentication foundation

Mon 22 Jun — Fri 3 Jul 2026

Backend-only sprint by design. The auth API gets built and verified before any mobile work starts — saves several weeks of rework when the mobile app later integrates against a stable backend in Sprint 2.

What we're building this sprint
Backend (sole focus this sprint)
  • NestJS application scaffold with logging, error handling, and OpenAPI / Swagger documentation
  • Shared infrastructure: idempotency, rate limiting, request tracing, structured error responses
  • User management endpoints (create, read, update, soft-delete)
  • OAuth providers — Google and Apple ID token verification with user upsert
  • JWT session tokens with rotating refresh tokens
  • Password hashing (argon2id) and reset flow
  • Phone OTP service with development-safe mock (real provider swaps in Sprint 5)
  • Email verification flow
  • Biometric token exchange endpoint (used by Sprint 2 mobile)
  • Role-based access control (Client / Tasker / Admin / Super-Admin)
  • Account suspension and ban endpoints
  • Session revocation (logout-from-all-devices)
  • Re-authentication gate for sensitive changes (change email, change phone)
  • Immutable audit log capturing every authentication and role-change event — enforced both at the application layer and at the database layer (the application database role has its UPDATE and DELETE permissions revoked on the audit-log table)
  • Integration tests covering happy path, unauthenticated, unauthorised, and replay-protection scenarios
Admin (gate only)
  • Admin login with session timeout (the full admin console arrives in Sprint 9)
Flutter Web — foundation (added per 14 Jun founder re-scope)
  • Flutter Web build target enabled + responsive layout primitives — same Flutter codebase, two target surfaces (mobile app + browser). Required before Sprint 2 can land web parity for each screen.
  • Web auth via HttpOnly cookie + CSRF — backend auth surface is cookie-aware; mobile uses Bearer tokens, web uses Set-Cookie + double-submit token. Same endpoints, dispatched by surface header. (Per ADR 006.)
  • Web routing — clean URLs and deep-link routes — go_router with setUrlStrategy(PathUrlStrategy()), browser back/forward respected, shareable URLs.

  • Swagger / OpenAPI documentation accessible and lists every endpoint
  • Database query shows every signup, login, refresh, and logout recorded in the audit log
  • OTP service uses a development mock that accepts code 000000; production safeguards visible in startup logs
  • Continuous integration is green on the merge commit
  • Security review checklist passes
  • Flutter Web shell loads in Chrome at localhost:5555, responsive layout breakpoints work, web auth round-trip succeeds via cookie + CSRF
  • Sprint 2 mobile-side mockups walked through so the client sees what's coming the following Friday
Deliberately not in this sprint (deferred):
  • Mobile app — starts Sprint 2 (against this now-stable backend)
  • Tasker upgrade flow + Stripe Connect — Sprint 2
  • Real production OTP provider — Sprint 5 (mock used in development for safety)

Sprint 3

Job posting + AI extraction + guest mode

Mon 20 Jul — Fri 7 Aug 2026 (3 weeks — extended to absorb AI infrastructure + Flutter Web parity + SEO phase 1)

The largest single AI sprint. Anyone, including a guest user with no account, can post a job; the AI does the heavy lifting of categorising, pricing, and clarifying.

What we're building this sprint
Mobile
  • Guest-mode shell with read-only browse of the public feed
  • Task-posting entry from anywhere in the app
  • Category picker
  • Title and free-text description
  • AI extraction confirmation screen (single-pass)
  • Multi-turn ReAct clarifying-question loop (when the AI needs more info)
  • Photo upload (multiple)
  • Camera-based task creation (vision model extracts task details from photos)
  • Voice-driven posting (voice → transcript → AI extraction)
  • Location and address picker on Google Maps
  • Date and time selection
  • Duration estimate (AI-suggested, user-editable)
  • Budget input with AI nudge
  • Special requirements field
  • Review-and-publish screen
  • Save-as-draft and resume-draft
  • Schedule jobs more than 7 days out
  • Edit and cancel a posted task
  • "Sign up to publish" bottom sheet for guests with local draft persistence
  • Auth-on-publish flow that transfers the draft to the new account
Backend
  • Task CRUD and state machine
  • AI extraction via Gemini Flash (JSON-mode for reliability)
  • ReAct multi-turn orchestration with confidence scoring
  • Multimodal vision extraction (Gemini Flash → Pro fallback if confidence low)
  • Image upload pipeline (local filesystem at MVP; Azure Blob in Sprint 10)
  • Geocoding via Google Maps Platform (server-side proxy to keep API key safe)
  • Lifecycle audit log of task changes
  • Vector embeddings generated on publish (used by ranked feed in Sprint 4)
  • Task draft persistence + claim-on-signup endpoint
  • Task search endpoint
  • Public read endpoint with PII filter and anonymous rate limiting
  • Voice transcription endpoint (Gemini audio)
Flutter Web parity (added per 14 Jun founder re-scope)
  • Guest browse (the highest-traffic web surface — shareable URLs, no auth required), posting entry, category picker, title + description form, AI extraction confirmation, ReAct clarifying-question loop on web
  • Photo upload (multi), camera-based posting (browser getUserMedia() on desktop, file picker fallback), voice path (browser MediaRecorder API)
  • Location/address picker (Google Maps JS API), date picker, budget, special requirements, review-and-publish, save-as-draft + resume on web
  • "Sign up to publish" modal + local draft persistence via browser localStorage + auth-on-publish flow — even more important on web than mobile (no app-store friction; makes shareable URLs convertible)
Next.js SEO public site — phase 1 (added per 14 Jun founder re-scope)
  • Public job detail page (Next.js, server-rendered, indexable) — URL /jobs/<id>-<slug>; uses the same public read endpoint as Flutter Web guest browse.
  • Schema.org structured data — JobPosting + LocalBusiness markup in JSON-LD so Google can render rich result cards in search.
  • Open Graph + Twitter card metadata per job — image (first photo or generated card), title, description, canonical URL. Drives sharing CTR on Facebook / WhatsApp / Twitter / LinkedIn.
  • Slug generation on job publish — backend populates Job.slug from title with collision handling.
AI infrastructure cluster (added per scope reconciliation)
  • LLM observability (Langfuse) — every external AI call is traced, with model, prompt version, tokens, latency, and cost recorded. Source of truth for AI behaviour and spend in production.
  • Prompt versioning and evaluation harness — every prompt lives in a versioned file; a continuous-integration evaluation runner checks new prompt versions against a fixture set and gates merges. Required for safe AI iteration.
  • AI cost-quota service — daily caps per user and globally. Vision and voice are roughly 10× the cost of text, so cost runaway is a real risk. Quota checks run before every AI call.
  • Output validators on every AI surface — schema validation across extraction, ReAct, vision, voice, future dispute mediator, and admin co-pilot. Single retry with corrective re-prompt; if still invalid, the surface falls back gracefully.
  • Prompt-injection defence — every place user-supplied text is embedded in a prompt goes through a hardening helper (delimiters, instruction reminder, control-character strip). Tested against the standard prompt-injection corpus.

  • Task search returns expected results
